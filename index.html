Method, constraints, and standards basis Source merge basis and invariants Two source assets were treated as authoritative: The most recent full dashboard HTML (structure, data and rendering logic). Internals were adjusted using CSS Grid/Flex layout within each panel but without changing the top-row track maths. CSS Grid is designed for explicit two-dimensional track layout and is appropriate for dashboards that require stable banding and alignment. Why explicit track sizing and “band” grids were prioritised Overlaps and cramped stacking in dashboards are usually caused by insufficiently constrained vertical tracks and inconsistent minimum sizing. In this merge, the forecast and tide internals were redesigned to use explicit grid rows (reserved heights) so each layer (icons, plot, bars, time axis) has guaranteed space. This aligns with the core grid model: define tracks explicitly and place items into cells for predictability. Why aspect-ratio was not relied upon for the webcam card In grid contexts, aspect-ratio influences auto sizing calculations and can create unexpected pressure on grid tracks. The webcam is instead kept “full-bleed” inside a fixed-width column derived from --top-row-height; this avoids the common grid sizing conflicts that can arise when multiple elements attempt to enforce independent aspect constraints.
